# Design: Contract-first Conductor Dispatch

**Date:** 2026-06-28
**Status:** Draft for review
**Branch:** `codex/subagent-support`

## Summary

V1 adds a Contract-first Conductor Dispatch layer for AWS worker agents.

V1 does not parse `SKILL.md` body text.
V1 does not read `workflow-schema.yaml` to drive task brief generation.
V1 does not use `.opencode/agents` as a truth source.
V1 only adds the `aws conductor *` command group and does not change existing `aws run`, `aws report inspect`, `aws gate check`, or `aws status` behavior.

The core path is:

```text
aws conductor brief/dispatch
  -> phase_mapping + role_policy + static_contracts
  -> task-brief.json + prompt.md
  -> runtime adapter
  -> task-result.json
  -> result validator
  -> events.ndjson
  -> execution-graph.json
```

Core boundaries:

```text
Conductor can generate tasks.
Worker agent can only execute tasks.
Gate remains the final truth.
OpenChamber only observes events/graph.
```

## Goals

- Define structured `task-brief`, `task-result`, event, graph, runtime, and path-policy contracts.
- Add deterministic phase-to-role mapping and role policies.
- Add `aws conductor` CLI commands for brief creation, dispatch, result validation, and graph rebuild.
- Add append-only events and a derived execution graph for OpenChamber.
- Add local and OpenCode command runtime adapters.
- Add OpenCode agent templates as repository-maintained assets.
- Preserve existing CLI behavior for run, report, gate, and status.
- Make dispatch auditable through both reported task results and actual git diff audit.

## Non-goals

- Do not implement an `omoRuntime`.
- Do not call `omo run`.
- Do not require or directly depend on OMO.
- Do not force worker agents to inherit or load skills.
- Do not put full `SKILL.md` content into worker prompts.
- Do not let worker agents modify `workflow-state.yaml`.
- Do not let worker agents run `aws gate check`.
- Do not let worker agents decide final PASS/FAIL.
- Do not strongly depend on OpenCode SDKs.
- Do not change existing `aws run`, `aws report inspect`, `aws gate check`, or `aws status` behavior.

### Truth sources

V1 task brief generation uses only:

- `phase_mapping`
- `role_policy`
- `static_contracts`
- explicit CLI options

V1 does not use these as brief-generation truth sources:

- `SKILL.md` body text
- `workflow-schema.yaml`
- `.opencode/agents/*.md`

`.opencode/agents` files are role prompt templates for OpenCode. They are not policy truth, phase mapping truth, or validation truth.

## Task brief contract

`TaskBrief` is the only task entrypoint for a worker agent. It is generated by Conductor and written to:

```text
qa/changes/<change-id>/orchestration/task-briefs/<task-id>.json
```

A companion prompt is written to:

```text
qa/changes/<change-id>/orchestration/prompts/<task-id>.md
```

The brief includes at least:

```ts
type TaskBrief = {
  schema_version: "1.0";
  task_id: string;
  change_id: string;
  session_id: string;
  parent_node_id?: string;
  node_id: string;

  phase: string;
  agent_role: string;

  title: string;
  goal: string;

  inputs: Array<{
    path: string;
    required: boolean;
    description?: string;
  }>;

  outputs: Array<{
    path: string;
    required: boolean;
    description?: string;
  }>;

  allowed_write_paths: string[];
  readonly_paths: string[];
  forbidden_actions: string[];
  quality_rules: string[];

  evidence_requirements: Array<{
    name: string;
    path: string;
    required: boolean;
  }>;

  command_policy: {
    allowed_commands: string[];
    forbidden_commands: string[];
    may_run_tests: boolean;
    may_run_aws_gate: false;
    may_modify_workflow_state: false;
  };

  timeout_ms: number;
  created_at: string;
  created_by: "aws-conductor";
};
```

Every brief must include the task result path in both `outputs` and `allowed_write_paths`:

```text
qa/changes/<change-id>/orchestration/task-results/<task-id>.json
```

This prevents the worker from being required to write `task-result.json` while the path policy forbids it.

The generated prompt is only a boundary reminder. It must not include `SKILL.md` body text. It must include:

```text
You are a bounded AWS worker agent.
Read the attached task-brief.json.
Execute only the task defined in the brief.
Write task-result.json to the required output path.
Respect allowed_write_paths.
Do not read SKILL.md as runtime instruction.
Do not modify workflow-state.yaml.
Do not run aws gate check.
Do not decide final PASS/FAIL.
```

## Task result contract

`TaskResult` is the worker agent's structured result:

```ts
type TaskResult = {
  schema_version: "1.0";
  task_id: string;
  change_id: string;
  node_id: string;
  agent_role: string;
  phase: string;

  status: "SUCCESS" | "FAILED" | "PARTIAL" | "BLOCKED";
  summary: string;

  files_created: string[];
  files_modified: string[];
  files_read?: string[];

  evidence: Array<{
    name: string;
    path: string;
    present: boolean;
  }>;

  findings?: Array<{
    type: string;
    severity: "info" | "warning" | "error";
    message: string;
    path?: string;
  }>;

  known_product_issue_candidates?: Array<{
    case_id?: string;
    endpoint?: string;
    description: string;
    evidence_path?: string;
  }>;

  policy_violations?: Array<{
    type: string;
    message: string;
    path?: string;
  }>;

  completed_at: string;
};
```

## Path policy and result validation

Task result validation must not rely only on worker-reported files.

For `opencode-command` runtime, Conductor captures a pre-dispatch git snapshot and a post-dispatch git snapshot. The validator checks both:

1. files reported in `task-result.json`
2. actual changed files detected from git diff and untracked files

Both sets must comply with `allowed_write_paths` and hard deny rules. Reported files should cover actual changed files.

Validation inputs:

```text
reported files = task-result.json files_created + files_modified
actual files   = git snapshot diff after excluding conductor-owned paths
```

Final validation rules:

- `task_id`, `change_id`, `phase`, and `agent_role` must match the brief.
- Reported files must be allowed.
- Actual changed files must be allowed.
- Actual changed files must be reported by the worker result.
- Required evidence must be present in the result and on disk.
- Hard deny paths always override allowlists.
- Any validation failure rejects the result.

### Changed file model

Git diff audit uses:

```ts
type ChangedFile = {
  path: string;
  previous_path?: string;
  status: "added" | "modified" | "deleted" | "renamed" | "untracked";
};
```

Audit rules:

- `added`, `modified`, and `untracked`: audit `path`
- `deleted`: audit deleted `path`
- `renamed`: audit both `previous_path` and `path`

Renamed files are audited on both sides so an agent cannot rename a forbidden file into an allowed path or an allowed file into a forbidden path.

### Required evidence

For each required entry in `brief.evidence_requirements`:

```text
required = true
  -> result.evidence must contain the same evidence name
  -> evidence.present must be true
  -> evidence.path must match the brief requirement
  -> evidence.path must not escape the repository
  -> evidence.path must exist on disk
```

The validator must not trust `present: true` without checking the filesystem.

### Hard deny paths

All paths are normalized to repo-relative POSIX paths. The path policy rejects empty paths, absolute paths, and paths containing `..` path traversal.

These hard deny paths always win over allowlists:

```text
workflow-state.yaml
qa/changes/<change-id>/workflow-state.yaml
qa/changes/<change-id>/execution/quality-gate-result.json
qa/changes/<change-id>/inspect/failure-analysis.json
```

For example, this must reject:

```text
allowed_write_paths = ["qa/changes/CHG-001/**"]
modified = ["qa/changes/CHG-001/execution/quality-gate-result.json"]
```

`aws-inspector-helper` may write:

```text
qa/changes/<change-id>/inspect/helper-findings.json
```

It must not overwrite:

```text
qa/changes/<change-id>/inspect/failure-analysis.json
```

`aws-fixer` has broad role capability, but each healing brief must narrow its actual writes to explicitly authorized files. The validator only trusts the final `allowed_write_paths` in the brief.

Example narrowed fixer brief:

```text
tests/api/test_menu.py
qa/changes/<change-id>/healing/**
qa/changes/<change-id>/orchestration/task-results/<task-id>.json
```

## Events and execution graph

Events are observability artifacts, not truth.

```text
events.ndjson          = append-only audit log
execution-graph.json   = derived UI cache
```

`events.ndjson` must not be cleaned, compressed, reordered, or rewritten by `graph_writer`. `graph_writer` only reads events and rebuilds graph. OpenChamber may read graph for UI, but debugging should use events as the source audit log.

Events are written to:

```text
qa/changes/<change-id>/orchestration/events.ndjson
```

Each line is one JSON event. Event types:

```text
session_started
phase_selected
task_brief_created
agent_dispatch_started
agent_dispatch_completed
agent_dispatch_failed
task_result_validated
task_result_rejected
gate_handoff
session_completed
```

Event shape:

```ts
type ConductorEvent = {
  schema_version: "1.0";
  event_id: string;
  session_id: string;
  change_id: string;
  node_id: string;
  parent_node_id?: string;
  phase?: string;
  agent_role?: string;
  type: ConductorEventType;
  status?: "PENDING" | "RUNNING" | "SUCCESS" | "FAILED" | "BLOCKED";
  message: string;
  artifact_paths?: string[];
  timestamp: string;
};
```

Key events must include `artifact_paths`:

- `task_brief_created`
- `agent_dispatch_started`
- `agent_dispatch_completed`
- `agent_dispatch_failed`
- `task_result_validated`
- `task_result_rejected`

The execution graph is written to:

```text
qa/changes/<change-id>/orchestration/execution-graph.json
```

Graph shape:

```ts
type ExecutionGraph = {
  schema_version: "1.0";
  session_id: string;
  change_id: string;
  root_node_id: string;
  nodes: Array<{
    id: string;
    parent_id?: string;
    type: "session" | "phase" | "agent" | "gate";
    label: string;
    phase?: string;
    agent_role?: string;
    status: "PENDING" | "RUNNING" | "SUCCESS" | "FAILED" | "BLOCKED";
    artifact_paths: string[];
    started_at?: string;
    completed_at?: string;
  }>;
  edges: Array<{
    from: string;
    to: string;
    type: "contains" | "depends_on" | "handoff";
  }>;
  updated_at: string;
};
```

`agent_dispatch_failed` and `task_result_rejected` are distinct:

- `agent_dispatch_failed`: runtime failed before producing a valid dispatch result, such as missing command template, OpenCode unavailable, timeout, non-zero exit, missing result file, or git audit unavailable.
- `task_result_rejected`: the worker ran and returned a result, but validation rejected it due to mismatched identity, path policy violation, missing evidence, hard-deny modification, or unreported actual changes.

## Runtime adapters

Runtime adapter only executes external workers.

It does not decide workflow phase.
It does not decide gate result.
It does not parse `SKILL.md`.

Interface:

```ts
export interface SubagentRuntimeAdapter {
  name: string;
  dispatch(input: {
    projectRoot: string;
    briefPath: string;
    promptPath: string;
    expectedResultPath: string;
    agentRole: string;
    changeId: string;
    taskId: string;
    timeoutMs: number;
    commandTemplate?: string;
  }): Promise<{
    status: "PENDING_EXTERNAL" | "COMPLETED" | "FAILED";
    exitCode?: number;
    stdoutPath?: string;
    stderrPath?: string;
    resultPath?: string;
    message?: string;
  }>;
}
```

### Local runtime

`local_runtime.ts` only prepares artifacts and returns:

```text
PENDING_EXTERNAL
```

It does not run an agent and does not perform actual diff audit.

### OpenCode command runtime

`opencode_command_runtime.ts` is strict. It must either run OpenCode successfully and produce `task-result.json`, or fail closed.

There is no built-in default command template in V1. Documentation provides a recommended template, but users must configure it in `.aws/config.yaml` or pass `--command-template`.

For `opencode-command` runtime:

- no template = `FAILED`
- OpenCode unavailable = `FAILED`
- non-zero exit = `FAILED`
- exit 0 but missing result = `FAILED`
- git audit unavailable = `FAILED`
- no silent fallback to local runtime

Supported command template variables:

```text
{{project_root}}
{{agent_role}}
{{brief_path}}
{{prompt_path}}
{{result_path}}
{{task_id}}
{{change_id}}
```

The implementation must avoid shell injection. By default, parse the command
template into command plus args and use:

```ts
child_process.spawn(command, args, { shell: false })
```

If the implementation cannot safely split the command template in V1, any shell-based fallback must be documented as trusted local config only and covered by tests. It must only interpolate normalized paths and known identifiers from the approved variable list above.

Recommended template for documentation:

```yaml
subagents:
  runtime: opencode-command
  timeout_ms: 900000
  command_template: >
    opencode run
    --dir {{project_root}}
    --agent {{agent_role}}
    --file {{brief_path}}
    --format json
    "Read the attached task-brief.json and execute only that task. Write task-result.json to {{result_path}}."
```

### Dispatch order for opencode-command

The order is fixed:

1. Conductor generates brief, prompt, and initial events.
2. Conductor records pre-dispatch git snapshot.
3. Conductor starts `opencode-command` runtime.
4. Conductor records post-dispatch git snapshot.
5. Conductor computes `actual_changed_files`.
6. Conductor validates `task-result.json`.
7. Conductor writes validation event and graph.

The pre-dispatch snapshot is taken after brief/prompt creation and before agent dispatch. This prevents Conductor-created brief and prompt files from being counted as worker changes.

During runtime, Conductor-owned paths are excluded from worker diff audit:

```text
qa/changes/<change-id>/orchestration/logs/<task-id>.stdout.log
qa/changes/<change-id>/orchestration/logs/<task-id>.stderr.log
qa/changes/<change-id>/orchestration/events.ndjson
qa/changes/<change-id>/orchestration/execution-graph.json
```

`task-result.json` is not excluded. It is a worker artifact and must be covered by `allowed_write_paths`.

## CLI commands

The new command group is registered in `src/cli.ts`:

```text
aws conductor brief
aws conductor dispatch
aws conductor validate-result
aws conductor graph
```

### `aws conductor brief`

Generates brief, prompt, events, and graph. Does not execute an agent.

```bash
aws conductor brief \
  --change CHG-001 \
  --phase api-codegen \
  --title "Generate API tests for menu module"
```

Options:

```text
--agent <role>
--session <session-id>
--parent-node <node-id>
--out <path>
--format json|pretty
--no-events
```

`--out` means orchestration output root, not a single brief file. For example:

```text
--out qa/changes/CHG-001/orchestration
```

The command still writes:

```text
task-briefs/<task-id>.json
prompts/<task-id>.md
events.ndjson
execution-graph.json
```

It reserves the expected task result path in the brief, but it does not create
`task-results/<task-id>.json`; that file is written by the worker agent.

Exit codes:

- success generating artifacts: `0`
- invalid phase or agent: `1`
- write or validation error: `1`

### `aws conductor dispatch`

Generates brief and prompt, calls runtime adapter, optionally validates result, and rebuilds graph.

```bash
aws conductor dispatch \
  --change CHG-001 \
  --phase api-codegen \
  --runtime opencode-command
```

Options:

```text
--runtime local|opencode-command
--command-template <template>
--timeout-ms <number>
--no-validate
--no-events
```

Exit codes:

- `--runtime local` artifacts generated and `PENDING_EXTERNAL`: `0`
- `--runtime opencode-command` command succeeds and result validates: `0`
- runtime failure, validation rejection, or git audit unavailable: `1`

### `aws conductor validate-result`

Offline reported-only validation:

```bash
aws conductor validate-result \
  --brief qa/changes/CHG-001/orchestration/task-briefs/<task-id>.json \
  --result qa/changes/CHG-001/orchestration/task-results/<task-id>.json
```

This command does not have actual diff audit by default. Its output and events must make that explicit:

```json
{
  "validation_mode": "reported_only",
  "diff_audit": "not_available"
}
```

Exit codes:

- valid: `0`
- invalid: `1`

### `aws conductor graph`

Rebuilds execution graph from events:

```bash
aws conductor graph --change CHG-001
```

Exit codes:

- graph rebuilt: `0`
- events missing or invalid: `1`

`.aws/config.yaml` may include:

```yaml
subagents:
  runtime: opencode-command
  timeout_ms: 900000
  command_template: >
    opencode run
    --dir {{project_root}}
    --agent {{agent_role}}
    --file {{brief_path}}
    --format json
    "Read the attached task-brief.json and execute only that task. Write task-result.json to {{result_path}}."
```

CLI options override config values.

No `omo run` command appears in config examples. If OMO is installed, it is loaded by OpenCode as a plugin. Conductor only knows that it is calling an OpenCode command template.

## OpenCode agent templates

OpenCode agents are repository template assets. Store them in:

```text
templates/opencode-agents/
  aws-explorer.md
  aws-case-designer.md
  aws-case-reviewer.md
  aws-plan-designer.md
  aws-plan-reviewer.md
  aws-api-builder.md
  aws-e2e-builder.md
  aws-fuzz-builder.md
  aws-performance-builder.md
  aws-fixer.md
  aws-inspector-helper.md
  aws-report-writer.md
```

They are copied to user projects as:

```text
<project>/.opencode/agents/*.md
```

Copying only happens when OpenCode integration is selected or detected. `aws init --repair` only repairs agent templates when OpenCode integration is detected or explicitly requested by an OpenCode repair path.

Copy rules:

- create missing agent files
- do not overwrite existing user agent files
- record skipped or warning for existing names
- repair mode also fills missing files

Each template uses `mode: all` so automation can run:

```bash
opencode run --agent aws-api-builder ...
```

Example frontmatter:

```yaml
---
name: aws-api-builder
mode: all
description: Generate pytest API tests from a bounded AWS task brief. Never run workflow gates.
---
```

Common body:

```text
You are a bounded AWS worker agent.
You do not own workflow state.
You do not read SKILL.md as runtime instruction.
You do not run aws gate check.
You do not decide final PASS/FAIL.
You must read the provided task-brief.json.
You must write task-result.json.
You must respect allowed_write_paths.
```

If the repository is packaged for npm, include:

```text
templates/opencode-agents/**
```

in package files or asset copy logic so installed users can access templates.

### Phase mapping tests

- `api-codegen -> aws-api-builder`
- `e2e-codegen -> aws-e2e-builder`
- `fuzz-codegen -> aws-fuzz-builder`
- `performance-codegen -> aws-performance-builder`
- `api-plan -> aws-plan-designer`

- non-agent phases such as `run`, `status`, and `gate-check` reject

### Brief builder tests

- brief contains correct role
- brief uses static contracts, not workflow schema
- task-result path is in outputs and allowed write paths
- `may_run_aws_gate === false`
- `may_modify_workflow_state === false`
- `aws-fixer` can be narrowed to explicit files

### Path policy tests

- allow `tests/api/test_x.py` for API builder
- reject absolute paths
- reject `../outside.txt`
- hard deny beats broad allowlist
- renamed audits old and new path
- deleted files are audited

### Result validator tests

- valid reported result passes
- mismatched `task_id` fails
- reported file outside allowlist fails
- actual file outside allowlist fails
- actual file not reported fails
- `workflow-state.yaml` modification fails
- `quality-gate-result.json` modification fails
- `failure-analysis.json` modification fails
- required evidence with `present: true` but missing file fails

### Events and graph tests

- event writer appends NDJSON
- graph writer only reads events and rebuilds graph
- events are not cleaned, compressed, or reordered
- `artifact_paths` exist on key events
- invalid event file makes graph command fail clearly

### Schema tests

- generated `TaskBrief` validates against `schemas/task-brief.schema.json`
- generated `TaskResult` validates against `schemas/task-result.schema.json`
- generated `ConductorEvent` validates against `schemas/conductor-event.schema.json`
- generated `ExecutionGraph` validates against `schemas/execution-graph.schema.json`

### Runtime tests

- local runtime returns `PENDING_EXTERNAL`
- opencode-command requires `command_template`
- opencode-command expands only approved variables
- opencode-command uses `spawn(command, args, { shell: false })` by default
- any shell-based fallback is documented as trusted local config only and covered by tests
- stdout and stderr logs are conductor-owned paths
- git snapshot unavailable fails closed
- missing result after exit 0 fails
- non-zero exit fails
- timeout fails

### Init and repair tests

- OpenCode integration path copies missing agent templates
- existing `.opencode/agents/*.md` files are not overwritten
- existing agent files are reported as skipped or warning
- `aws init --repair` fills missing agent templates when OpenCode integration is detected or requested
- non-OpenCode integration path does not copy agent templates

### CLI smoke tests

- `aws conductor brief --change TEST --phase api-codegen` exits 0
- generated brief, prompt, events, and graph exist
- `aws conductor dispatch --runtime local` exits 0 with `PENDING_EXTERNAL`
- `aws conductor validate-result` valid exits 0
- `aws conductor validate-result` invalid exits 1
- `validate-result` output includes `validation_mode=reported_only` and `diff_audit=not_available`
- `aws conductor graph --change TEST` exits 0 when events are valid
- opencode-command failure paths exit 1
- CLI commands return deterministic exit codes:
  - valid brief, local dispatch, valid result, and valid graph return `0`
  - invalid phase, invalid result, runtime failure, and invalid events return `1`
